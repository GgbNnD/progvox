好的，这是一个非常详尽的课程项目方案。为了便于管理和评估，我将其核心目标——“实现一个生成式极低码率视频通话原型系统”——按照方案中的四个阶段，分解为更细粒度、可执行、可度量的小任务，并为每个任务明确了具体的可交付成果。

这份任务分解清单可以直接作为项目管理看板使用。

### 第一阶段：环境搭建与学习型编码器基础（第1-3周）

此阶段的核心风险是开发环境和数据集是否就绪，基线是否建立。任务分解如下：

**1.1 任务：基础开发与实验环境搭建**
*   **任务描述**：完成所有软硬件环境的安装与验证，确保能跑通一个基础的AI视频压缩模型。
*   **可交付成果**：
    1.  **环境验证截图**：终端截图显示成功执行 `python -c "import torch; print(torch.__version__); import compressai; print(compressai.__version__)"`，输出版本号且无报错。
    2.  **Demo脚本运行记录**：终端日志，显示成功运行 CompressAI 官方提供的 `codec.py` 示例脚本，对一个测试视频完成编码和解码。
    3.  **`environment.yml` 或 `requirements.txt` 文件**：包含所有已安装依赖及其版本的配置文件，确保环境可复现。

**1.2 任务：核心数据集准备与验证**
*   **任务描述**：下载项目必需的主要数据集，并进行初步的格式和内容校验。
*   **可交付成果**：
    1.  **数据集目录结构文档**：一个 `dataset_readme.md` 文件，截图或文本展示 `data/` 下的目录结构，清晰分好 VoxCeleb2（少量样本）、UVG、Xiph-5N 等子目录。
    2.  **数据样本分析脚本与日志**：一个 `dataset_stats.py` 脚本，运行后能打印出各数据集的视频数量、分辨率分布、总时长等统计信息。

**1.3 任务：Baseline学习型视频编码器(LVC)运行与评测**
*   **任务描述**：使用 CompressAI 的预训练模型 SSF2020，对标准测试视频进行编码，并计算关键客观质量指标。
*   **可交付成果**：
    1.  **预处理脚本**：`prepare_test_videos.py`，用于将原始测试集视频裁切成固定长度的片段，统一为编码器要求的格式。
    2.  **自动化评测脚本**：`eval_lvc_baseline.sh` (或Python脚本)，该脚本调用 CompressAI 工具，对一个给定的测试视频文件夹，在4个不同质量等级下运行编解码，并自动收集每个点的 码率(Bitrate)、PSNR、MS-SSIM。
    3.  **Baseline性能数据 (CSV文件)**：`ssf2020_rd_points.csv`，包含测试集中每个视频在不同质量点下的压缩性能数据。

### 第二阶段：生成式压缩核心实现（第4-8周）

此阶段是技术攻坚核心，将任务分解为三个独立的模块进行开发和验证，最后再进行系统集成评估。可交付成果是代码、模型权重和实验报告。

**2.1 任务：ProGVC 多尺度 Token 化器开发与验证**
*   **任务描述**：实现视频帧的多尺度残差金字塔分解和矢量量化，能将一帧图像压缩为一组分层 token maps。
*   **可交付成果**：
    1.  **`models/tokenizer.py` 模块代码**：PyTorch模型代码，包含完整的多尺度分解和码本量化逻辑，代码注释完整。
    2.  **单元测试通过报告**：`tests/test_tokenizer.py` 的运行日志，证明对于给定输入，Tokenizer 能正确输出 K=4 个不同尺度的 token map，且通过反量化能重建出不同清晰度的图像。
    3.  **Token化效率分析报告 (Jupyter Notebook)**：`analysis_tokenizer.ipynb`，对50张测试图片，分析了在不同截断尺度下（只保留前K层token），重建图像的SSIM和Token数据量（以kbps为单位），并附有视觉对比图。

**2.2 任务：细节合成生成器开发与消融实验**
*   **任务描述**：开发并训练至少两种生成器（CNN+GAN 和 轻量Diffusion），建立统一评测基准，以确定最终方案。
*   **可交付成果**：
    1.  **生成器模块代码与配置**：`models/generator.py` (包含CNN和Diffusion两种实现) 和 `train/train_generator.yaml` (训练超参数配置)。
    2.  **训练日志与模型权重**：TensorBoard 训练日志文件，以及最终选出的两个生成器的模型权重文件 (`.pth`)。
    3.  **生成器消融实验报告**：`docs/generator_ablation.md`，包含一个表格，横向对比“纯CNN”、“CNN+GAN”、“轻量Diffusion(4步)”三种方案在验证集上的PSNR, SSIM, LPIPS, FID指标，并有清晰的分析结论指出最终选择的方案及其理由。

**2.3 任务：自回归上下文模型与完整ProGVC链路集成**
*   **任务描述**：实现 Transformer 自回归模型，并打通“编码器 -> 熵编码 -> 模拟截断 -> 自回归预测 -> 生成器解码”的完整数据链路。
*   **可交付成果**：
    1.  **上下文模型模块代码**：`models/context_model.py`。
    2.  **端到端编解码脚本**：`run_progvc_codec.py`，接受一个视频和模拟的带宽限制（例如“只传Layer 0+1”），输出解码后的视频。
    3.  **全链路集成测试报告**：`docs/progvc_integration_test.md`，包含对完整 ProGVC 模型的性能初步测试结果。核心是一个对比表格，展示在总计约500kbps的带宽下，本项目ProGVC方案与SSF2020在PSNR和LPIPS上的对比。

### 第三阶段：渐进式传输与自适应码率（第9-11周）

此阶段将静态压缩算法变为动态、网络感知的系统，核心是调度器和控制器的逻辑。

**3.1 任务：分层Token调度器与网络模拟实现**
*   **任务描述**：在发送端实现一个基于优先级的 token 调度器，并能在模拟网络环境下工作。
*   **可交付成果**：
    1.  **调度器模块代码**：`transport/scheduler.py`。
    2.  **网络模拟测试环境搭建指南**：`docs/network_emu_guide.md`，详细说明如何使用 `mahimahi` 或 `tc netem` 复现一个有带宽波动和丢包的弱网环境。
    3.  **调度器性能对比报告**：`docs/scheduler_comparison.md`，在同一个模拟弱网trace下，对比“贪婪调度”和“滑动窗口调度”两种策略下，接收端解码视频的平均PSNR波动和卡顿率。

**3.2 任务：自适应码率（ABR）控制器开发**
*   **任务描述**：实现一个能根据网络状态（RTT, 丢包率）动态决定传输 token 最大尺度的决策逻辑。
*   **可交付成果**：
    1.  **ABR控制器模块代码**：`models/abr_controller.py`，至少实现基于规则（Rule-based）的控制器。
    2.  **ABR决策仿真验证 (Jupyter Notebook)**：`analysis_abr.ipynb`，使用 trace-driven 的方法，将一段网络带宽波动 trace 作为输入，画出ABR控制器输出的“传输尺度”随时间变化的曲线，与带宽曲线对比，验证其逻辑是否符合预期。

### 第四阶段：WebRTC集成与端到端系统联调（第12-15周）

此阶段将系统从离线处理变成实时通话原型，关键是打通 WebRTC 信令和数据通道，并完成最终演示和报告。

**4.1 任务：Janus WebRTC 服务部署与信令联通**
*   **任务描述**：成功部署 Janus Gateway，并实现浏览器或 Python 客户端之间的基础 WebRTC 视频通话。
*   **可交付成果**：
    1.  **部署文档**：记录 Janus 安装步骤、配置文件修改要点的文档。
    2.  **基础通话演示视频**：一个15秒的录屏，展示两个 `aiortc` 客户端（或浏览器客户端）通过 Janus 成功建立连接，并能看到对方摄像头的画面。
    3.  **信令流程图**：一张 Visio 或 Draw.io 绘制的流程图，清晰展示客户端与 Janus 之间创建会话、建立 PeerConnection 和 DataChannel 的信令交互顺序。

**4.2 任务：自定义 DataChannel 协议与传输实现**
*   **任务描述**：在 WebRTC DataChannel 之上，实现自定义的 token 二进制传输协议。
*   **可交付成果**：
    1.  **协议模块代码**：`transport/datachannel_proto.py`，包含数据包打包和解包的具体实现。
    2.  **协议单元测试通过报告**：`tests/test_protocol.py` 日志，验证打包和解包逻辑的正确性，以及能正确处理包乱序和超时的情况。
    3.  **传输鲁棒性测试报告**：记录在不同模拟丢包率（如 1%, 3%, 5%）下，接收端接收到的完整帧比例和解码成功率。

**4.3 任务：端到端系统联调、优化与交付**
*   **任务描述**：将所有模块集成到最终的 `sender_main.py` 和 `receiver_main.py` 中，进行系统级性能优化，并完成最终的项目成果交付。
*   **可交付成果**：
    1.  **最终版源代码仓库**：一个结构清晰、包含 `README.md` 的完整项目目录，任何人在满足环境要求下都能尝试运行。
    2.  **系统演示视频**：最终的 3-5 分钟高清演示视频。
    3.  **最终课程报告**：`docs/final_report.md`。
    4.  **系统性能剖析图/表**：在最终报告中，清晰展示编码延迟、生成延迟、端到端延迟、显存占用等指标，并与项目目标进行对比。
